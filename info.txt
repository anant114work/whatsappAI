/templates
Create Template Message


Use this endpoint to create WhatsApp template messages for reaching out to users.

Template messages can be type of:

Text Message
Media Templates
Interactive Templates
Location Templates
Multi-Product Templates
Catalog Templates
Order Details Templates
Checkout Button Templates
Order Status Templates
Coupon Code Templates
Carousel Templates
Form Templates
Limited-Time Offer Templates
Authentication Templates
Voice Call Templates
Call Permission Request Templates
Text Message

Send simple text templates to users or you can send personalized template messages using the variables

Media Templates

Here you can send Media Templates, which can be Video, Image or Document.

Interactive Templates

Here you can send the templates with Buttons. There are two types of predefined buttons:

Call-to-Action — Allows your customer to call a phone number and visit a website.
Quick Reply — Allows your customer to return a simple text message.
These buttons can be attached to text messages or media messages.

Location Templates

Location headers appear as generic maps at the top of the template and are useful for order tracking, delivery updates, ride hailing pickup/dropoff, locating physical stores, etc. You can share the location to users using the lattitude and longitude of the location.

Multi-Product Templates

MPM templates can be used to open marketing conversations. They allow you to showcase up to 30 products from your ecommerce catalog, organized in up to 10 sections, in a single message.

Catalog Templates

Catalog templates are marketing templates that allow you to showcase your product catalog entirely within WhatsApp. Catalog templates display a product thumbnail header image of your choice and custom body text, along with a fixed text header and fixed text sub-header.

Order Details Templates

Order details message template is one of the interactive message template that extends the call-to-action button to support sending order details as template and provides richer experience.

Checkout Button Templates

Checkout button templates are marketing templates that can showcase one or more products along with corresponding checkout buttons that WhatsApp users can use to make purchases without leaving the WhatsApp client.

Order Status Templates

Order status template allows you to provide order status updates to customers, such as processing, shipped, or completed, ensuring clear and timely communication with customers.

Coupon Code Templates

Coupon code templates are marketing templates that display a single copy code button. When tapped, the code is copied to the customer's clipboard.

Carousel Templates

Carousel templates allow you to send a single text message accompanied by a set of up to 10 carousel cards in a horizontally scrollable view.

Form Templates

Form Template enables businesses to offer a richer, more engaging user experience that can help customers get things done faster on WhatsApp without necessarily needing to switch to another app, or visit a website.

Limited-Time Offer Templates

Limited-time offer templates allow you to display expiration dates and running countdown timers for offer codes in template messages, making it easy for you to communicate time-bound offers and drive customer engagement.

Authentication Templates

Copy code authentication templates allow you to send a one-time password or code along with a copy code button to your users.

Voice Call Templates

Voice Call Templates allow WhatsApp consumers to initiate a voice call directly by clicking the button in the template.

Call Permission Request Templates

Call Permission Request Templates allow you to create templates that enable WhatsApp users to grant consent for receiving calls.

Note: For all Marketing type templates, you can set a Time-to-Live (TTL) value to define the delivery window, only if you have onboarded the MM Lite API. TTL must be specified in seconds, with a minimum of 43,200 seconds (12 hours) and a maximum of 2,592,000 seconds (30 days). The recommended TTL value is 86,400 seconds (24 hours).

Request Header

Key	Description
Authorization	Here you need to add the access token for the Authorization. You can copy the Access Token from here .
Request Body

Key	Description
to	Message recipient number. Must be in international format that includes country code. For example: +919090909090. Country code is must to add.
type	Type of message. Value should be template
template	The template object to build a message that will be sent
name	It is template name. You can copy the template name from the Template manager that you want to send. example: hello_world
language	It is a language code for the template. You can check the different language codes from here. Use the same language code that used while registering the template.
components	It's a list of components that is created during the template creation like header, body Interactive buttons.
components => type	Components types are body, button, header.
subtype	Required when type=button. Not used for the other types.
quick_reply: Refers to a previously created quick reply button that allows for the customer to return a predefined message.
url: Refers to a previously created button that allows the customer to visit the URL generated by appending the text parameter to the predefined prefix URL in the template.
index	Number of button index starting from 0, only if quick reply or dynamic URL buttons have been registered, It can have up to three quick reply buttons or only one dynamic URL button.
example: 0
minimum: 0
maximum: 2
parameters	This is required in case of a template registered with the variables.
parameters => type	Parameter types are image, video, document,location, payload
image => link	URL of a image sent in the header, only if having media URL. It is expected to be a valid URL starting with https:// or http://. Supported image types are PNG, JPG, JPEG. Maximum image size is 5MB.
image => id	This is the id of a media, You can create id by uploading the image using the media APIs explained in this document. Use the media id fetched after uploading the media.
video => link	URL of a video sent in the header, only if having media URL. It is expected to be a valid URL starting with https:// or http://. Supported video types are MP4, 3GPP. Maximum video size is 16MB.
video => id	This is the id of a media, You can create id by uploading the video using the media APIs explained in this document. Use the media id fetched after uploading the media.
document => link	URL of a document sent in the header, only if having media URL. It is expected to be a valid URL starting with https:// or http://. Supported document type is PDF. Maximum document size is 100MB
document => id	This is the id of a media, You can create id by uploading the document using the media APIs explained in this document. Use the media id fetched after uploading the media.
document => filename	Add a file name to show in WhatsApp.
location => latitude	Lattitude of the location. For example: 23.0917101.
location => longitude	Longitude of the location. For example: 72.5949363.
location => name	Name of the place. For example: Fab India Store.
location => address	Detailed address of the location For example: Mumbai Metropolitan Region
payload	Payload of a quick reply button, only if quick reply buttons have been registered. It should be in text format. Add the keyword as payload.
source	Source can be a label or string that indicates the platform or third party name from where the message is sent.
metaData => custom_callback_data	custom_callback_data optional is field allows businesses to pass a custom tracking value which will be returned unprocessed in all subsequent webhook callbacks (sent, delivered, read, failed).

Opt-in/ Opt-out


Use this endpoint to get the opt-ins and opt-out of the users to sent the messages on WhatsApp.

To start receiving messages on WhatsApp, users need to provide explicit consent through a third-party channel. This channel can include various communication methods like your website, mobile app, missed call, IVR, email, SMS, retail location, contact center, and WhatsApp session-based messages.

The opt-in process must be explicit, requiring a user action, such as entering a phone number or selecting a checkbox to confirm their consent.

Ensure transparent opt-in messaging so that users are informed about the types of messages they are subscribing to.

Businesses are responsible for maintaining opt-ins and should be prepared to provide this information if requested by WhatsApp.

Use this endpoint to send session messages to the users. This API helps you to reach out the users with WhatsApp free-form session messages.

Session messages can be type of:

Text Message

Send simple text messages to users.

Media Messages

Interactive Messages

WhatsApp provides various Interactive Messages using which you can share the Rich Media contetnt to your users and start easy interaction on WhatsApp. Here are the various types of Interactive Messages:

Message with Reply Button — Where you can send a message with max 3 buttons on WhatsApp, user will click the button and start interaction with business.
List Message — Here you can send a message with llist of options(max allowed 10). This can be used when a business wants to share a menu or various service on WhatsApp.
WhatsApp Forms — Here, businesses can define, configure, and customize messages with rich interactions that give customers more structure in the way they communicate.
Order Details — In this, businesses can send customers a bill, so the customer can complete their order by paying the business without having to leave WhatsApp.
Location Messages

Location Message — In this, location appear as generic maps at the top of the message and are useful for order tracking, delivery updates, ride hailing pickup/dropoff, locating physical stores, etc. You can share the location to users using the lattitude and longitude of the location.
Location Request — In this, you can ask to share the location on WhatsApp, where user can pin their location on WhatsApp or can share the current location with business.

/whatsapp-cloud/calls
Send Call Updates


Send Call Updates
This API allows you to control and update the state of WhatsApp calls. It supports initiating outbound calls, accepting or rejecting incoming calls, and terminating ongoing sessions.

🔄 Supported Call Actions
Action	Description	Required Fields
connect	Initiate an outgoing call	to, session.sdp_type = "offer", session.sdp
pre_accept	Pre-accept an incoming call (ICE warm-up)	call_id, session.sdp_type = "answer", session.sdp
accept	Accept an incoming call	call_id, session.sdp_type = "answer", session.sdp
reject	Reject an incoming call	call_id
terminate	Terminate an ongoing call	call_id
📤 Request Payload Format
The payload must include the messaging product type (whatsapp) and vary in required fields based on the action.

For outgoing calls, provide the to phone number and an SDP offer in the session object.
For incoming call responses, provide the call_id and an SDP answer (for accept/pre_accept).
For reject or terminate, only the call_id is necessary.
📥 SDP Exchange via session
The session object includes WebRTC SDP data:

sdp_type: Indicates if the payload is an SDP "offer" or "answer".
sdp: The SDP string (must be properly escaped sdp).
Example:

"session": {
  "sdp_type": "offer",
  "sdp": "{...SDP string...}"
}
📬 Response Format
For outgoing calls (action: connect), you will receive the following response:
{
  "calls": [
    {
      "id": "wacid.ABGGFjFVU2AfAgo6V-Hc5eCgK5Gh"
    }
  ]
}
For all other call actions (pre_accept, accept, reject, terminate), the response will be:
{
  "messaging_product": "whatsapp",
  "success": true
}

Create Template Message

Use this endpoint to create WhatsApp template messages for reaching out to users.

Text Message

Send simple text templates to users or you can send personalized template messages using the variables

Media Templates

Here you can send Media Templates, which can be Video, Image or Document.

Interactive Templates

Here you can send the templates with Buttons. There are two types of predefined buttons:

Call-to-Action — Allows your customer to call a phone number and visit a website.

Quick Reply — Allows your customer to return a simple text message.

These buttons can be attached to text messages or media messages.

Location Templates

Location headers appear as generic maps at the top of the template and are useful for order tracking, delivery updates, ride hailing pickup/dropoff, locating physical stores, etc. You can share the location to users using the lattitude and longitude of the location.

Multi-Product Templates

MPM templates can be used to open marketing conversations. They allow you to showcase up to 30 products from your ecommerce catalog, organized in up to 10 sections, in a single message.

Catalog Templates

Catalog templates are marketing templates that allow you to showcase your product catalog entirely within WhatsApp. Catalog templates display a product thumbnail header image of your choice and custom body text, along with a fixed text header and fixed text sub-header.

Order Details Templates

Order details message template is one of the interactive message template that extends the call-to-action button to support sending order details as template and provides richer experience.

Checkout Button Templates

Checkout button templates are marketing templates that can showcase one or more products along with corresponding checkout buttons that WhatsApp users can use to make purchases without leaving the WhatsApp client.

Order Status Templates

Order status template allows you to provide order status updates to customers, such as processing, shipped, or completed, ensuring clear and timely communication with customers.

Coupon Code Templates

Coupon code templates are marketing templates that display a single copy code button. When tapped, the code is copied to the customer's clipboard.

Carousel Templates

Carousel templates allow you to send a single text message accompanied by a set of up to 10 carousel cards in a horizontally scrollable view.

Form Templates

Form Template enables businesses to offer a richer, more engaging user experience that can help customers get things done faster on WhatsApp without necessarily needing to switch to another app, or visit a website.

Limited-Time Offer Templates

Limited-time offer templates allow you to display expiration dates and running countdown timers for offer codes in template messages, making it easy for you to communicate time-bound offers and drive customer engagement.

Authentication Templates

Copy code authentication templates allow you to send a one-time password or code along with a copy code button to your users.

Voice Call Templates

Voice Call Templates allow WhatsApp consumers to initiate a voice call directly by clicking the button in the template.

Call Permission Request Templates

Call Permission Request Templates allow you to create templates that enable WhatsApp users to grant consent for receiving calls.

Note: For all Marketing type templates, you can set a Time-to-Live (TTL) value to define the delivery window, only if you have onboarded the MM Lite API. TTL must be specified in seconds, with a minimum of 43,200 seconds (12 hours) and a maximum of 2,592,000 seconds (30 days). The recommended TTL value is 86,400 seconds (24 hours).

Send Call Updates

This API allows you to control and update the state of WhatsApp calls. It supports initiating outbound calls, accepting or rejecting incoming calls, and terminating ongoing sessions.

🔄 Supported Call Actions

Action  Description Required Fields

connect Initiate an outgoing call to, session.sdp_type = "offer", session.sdp

pre_accept  Pre-accept an incoming call (ICE warm-up) call_id, session.sdp_type = "answer", session.sdp

accept  Accept an incoming call call_id, session.sdp_type = "answer", session.sdp

reject  Reject an incoming call call_id

terminate Terminate an ongoing call call_id

📤 Request Payload Format

The payload must include the messaging product type (whatsapp) and vary in required fields based on the action.

For outgoing calls, provide the to phone number and an SDP offer in the session object.

For incoming call responses, provide the call_id and an SDP answer (for accept/pre_accept).

For reject or terminate, only the call_id is necessary.

📥 SDP Exchange via session

The session object includes WebRTC SDP data:

sdp_type: Indicates if the payload is an SDP "offer" or "answer".

sdp: The SDP string (must be properly escaped sdp).

Example:

"session": {

  "sdp_type": "offer",

  "sdp": "{...SDP string...}"

}

📬 Response Format

For outgoing calls (action: connect), you will receive the following response:

{

  "calls": [

    {

      "id": "wacid.ABGGFjFVU2AfAgo6V-Hc5eCgK5Gh"

    }

  ]

}

For all other call actions (pre_accept, accept, reject, terminate), the response will be:

{

  "messaging_product": "whatsapp",

  "success": true

}

To make all your WhatsApp and RCS messages (new and previous ones) appear on your CRM and allow you to reply to them, you'll need to implement an integration strategy that leverages the provided APIs and webhooks.

Here's a breakdown of the steps and concepts involved:

Understanding the Core Components

WhatsApp Voice Call APIs: This API set focuses on managing WhatsApp voice calls (initiating, accepting, rejecting, terminating) and does not directly handle text messages. The /whatsapp-cloud/calls endpoint and the associated connect, pre_accept, accept, reject, and terminate actions are for call control, not messaging. The settings endpoints (/whatsapp-cloud/settings) are for configuring call icon visibility and status.

RCS API: This API is for sending various types of RCS messages (/rcs/messages).

Webhooks (for both WhatsApp and RCS): This is the crucial part for getting incoming messages into your CRM. Webhooks allow the messaging platforms to send real-time notifications to your system whenever an event occurs (like a new incoming message, message status updates, etc.).

High-Level Integration Strategy

The general approach is to:

Set up your CRM to receive data: Your CRM needs to have a way to ingest incoming messages. This usually involves:

Custom API endpoints in your CRM: If your CRM supports it, you can create API endpoints that your webhooks will call.

Middleware/Integration Platform: Use a tool like Zapier, Make (formerly Integromat), n8n, or a custom application layer that acts as a bridge between the messaging platform webhooks and your CRM's API.

Configure Webhooks in the Messaging Platforms: Tell WhatsApp and RCS where to send notifications for incoming messages and status updates.

Process Incoming Messages: When a webhook notification is received by your CRM (or middleware), parse the message content, identify the sender, and create/update a corresponding record (e.g., a "conversation" or "activity" tied to a contact) in your CRM.

Enable Replying from CRM:

CRM UI: Design your CRM's interface to allow agents to type and send replies.

API Calls: When an agent sends a reply from the CRM, your CRM (or middleware) will then make an API call to the respective WhatsApp or RCS Send Message API to send the outgoing message.

Detailed Steps and Considerations

A. For WhatsApp Messages (assuming you're using the WhatsApp Business Platform, which is usually integrated with such APIs):

Incoming Messages (New and Previous):

Webhook Configuration: You'll need to set up webhooks for your WhatsApp Business Account (WABA). The Meta for Developers documentation (which these APIs likely leverage) outlines how to do this. You'll subscribe to events like messages to receive inbound messages.

Webhook Endpoint: Create an HTTPS endpoint on your server or within your middleware that can receive POST requests from WhatsApp's webhooks.

Parsing Webhook Payload: When a new message comes in, WhatsApp will send a JSON payload to your webhook endpoint. This payload will contain the message content, sender's phone number, message ID, and other relevant information.

CRM Integration:

Identify Contact: Use the sender's phone number from the webhook payload to look up an existing contact in your CRM. If no contact exists, you might want to create a new lead/contact.

Create/Update Conversation: Create a new "conversation" record or an "activity" log item linked to the contact in your CRM. Store the message content, timestamp, and message ID.

Display in CRM: Ensure your CRM's UI is designed to display these conversation threads clearly for your agents.

Replying to WhatsApp Messages from CRM:

CRM UI for Replies: Within your CRM, provide a text input field and a "Send" button associated with the WhatsApp conversation.

Outbound API Call: When an agent sends a reply:

Your CRM (or middleware) will need to call the WhatsApp Business API's "Send Message" endpoint. This is not explicitly detailed in the provided "WhatsApp Voice Call APIs" document, as that focuses on voice calls. You would typically use a separate WhatsApp Business Platform Message API for sending text messages.

The request payload would include the recipient's phone number (from the original incoming message), the message content, and potentially other parameters like message type (text, media, template).

Authorization: Remember to include your AUTH_TOKEN in the header of your API requests.

B. For RCS Messages:

Incoming Messages (New and Previous):

Webhook Configuration: The provided RCS API documentation explicitly mentions "Receive User Messages" and "Receive RCS Callbacks" webhooks. You'll need to configure your webhook URL in the RCS platform.

Webhook Endpoint: Similar to WhatsApp, create an HTTPS endpoint on your server or within your middleware to receive POST requests from the RCS webhooks.

Parsing Webhook Payload: The "Receive User Message" webhook example shows a payload including userPhoneNumber, botId, messageId, sendTime, and text. This is the data you'll need to parse.

CRM Integration:

Identify Contact: Use the userPhoneNumber to find or create a contact in your CRM.

Create/Update Conversation: Log the incoming RCS message as a conversation or activity linked to the contact in your CRM, storing all relevant details from the payload.

Display in CRM: Make sure your CRM's interface can show these RCS message threads alongside other communication.

Replying to RCS Messages from CRM:

CRM UI for Replies: Implement a mechanism in your CRM for agents to compose and send RCS replies.

Outbound API Call: When an agent sends a reply from the CRM:

Your CRM (or middleware) will call the /rcs/messages endpoint.

The request payload will include the to phone number, the content (e.g., plainText), and source.

Authorization: Include your AUTH_TOKEN in the header.

C. Handling Previous Messages (Historical Data):

Retrieving Past Conversations: The provided API documentation for both WhatsApp Voice Call and RCS does not include endpoints for retrieving historical message data. This is a common limitation with messaging APIs, where webhooks primarily provide real-time updates for new events.

Solution for Historical Data:

Initial Sync (if supported by the platform): Some WhatsApp Business Solution Providers (BSPs) or RCS platforms might offer a way to export historical chat logs, which you could then import into your CRM. However, this is not standard across all APIs.

Building a Historical Archive: As soon as you integrate the webhooks, you will start capturing all new incoming messages. For previous messages, if the platform doesn't offer a historical export, you might not be able to pull them directly through the API. Your best bet would be to ensure all future conversations are logged in your CRM once the integration is live.

Key Considerations for Implementation:

Authentication: Ensure your AUTH_TOKEN is securely managed and included in all API requests.

Error Handling: Implement robust error handling for both inbound webhook processing and outbound API calls. The API responses show various error codes (400, 401, 424) that you need to account for.

Rate Limits: Be aware of any rate limits imposed by the WhatsApp and RCS platforms to avoid getting blocked.

Scalability: Design your webhook endpoint and CRM integration to handle a high volume of messages if you anticipate a lot of customer interactions.

Message Types: The RCS API supports various message types (plain text, cards, carousels, etc.). Your CRM interface should ideally support sending and displaying these rich message formats for a better customer experience. WhatsApp also supports various message types beyond plain text (media, location, contacts, etc.), and your webhook processing should be able to handle these.

SDP Exchange for Voice Calls (WhatsApp): If you intend to use the WhatsApp Voice Call APIs, remember that session.sdp_type and session.sdp are critical for WebRTC Session Description Protocol (SDP) exchange, which is complex and requires specialized WebRTC knowledge to implement a functional voice call system.

CRM Capabilities: Your CRM needs to have the flexibility to integrate with external APIs, create custom objects (e.g., for conversations), and provide a user interface that can display and allow interaction with these messaging channels.

Consent and Compliance: Always ensure you comply with data privacy regulations (e.g., GDPR, CCPA) and the messaging platform's policies (WhatsApp Business Policy, RCS Business Messaging Guidelines) regarding user consent for messaging.

By carefully planning your webhook infrastructure and CRM integration points, you can effectively centralize your WhatsApp and RCS communications within your CRM for a unified customer view and efficient agent responses.Skip to main content

Text Message

Send simple text templates to users or you can send personalized template messages using the variables

Media Templates

Here you can send Media Templates, which can be Video, Image or Document.

Interactive Templates

Here you can send the templates with Buttons. There are two types of predefined buttons:

Call-to-Action — Allows your customer to call a phone number and visit a website.
Quick Reply — Allows your customer to return a simple text message.
These buttons can be attached to text messages or media messages.

Location Templates

Location headers appear as generic maps at the top of the template and are useful for order tracking, delivery updates, ride hailing pickup/dropoff, locating physical stores, etc. You can share the location to users using the lattitude and longitude of the location.

Multi-Product Templates

MPM templates can be used to open marketing conversations. They allow you to showcase up to 30 products from your ecommerce catalog, organized in up to 10 sections, in a single message.

Catalog Templates

Catalog templates are marketing templates that allow you to showcase your product catalog entirely within WhatsApp. Catalog templates display a product thumbnail header image of your choice and custom body text, along with a fixed text header and fixed text sub-header.

Order Details Templates

Order details message template is one of the interactive message template that extends the call-to-action button to support sending order details as template and provides richer experience.

Checkout Button Templates

Checkout button templates are marketing templates that can showcase one or more products along with corresponding checkout buttons that WhatsApp users can use to make purchases without leaving the WhatsApp client.

Order Status Templates

Order status template allows you to provide order status updates to customers, such as processing, shipped, or completed, ensuring clear and timely communication with customers.

Coupon Code Templates

Coupon code templates are marketing templates that display a single copy code button. When tapped, the code is copied to the customer's clipboard.

Carousel Templates

Carousel templates allow you to send a single text message accompanied by a set of up to 10 carousel cards in a horizontally scrollable view.

Form Templates

Form Template enables businesses to offer a richer, more engaging user experience that can help customers get things done faster on WhatsApp without necessarily needing to switch to another app, or visit a website.

Limited-Time Offer Templates

Limited-time offer templates allow you to display expiration dates and running countdown timers for offer codes in template messages, making it easy for you to communicate time-bound offers and drive customer engagement.

Authentication Templates

Copy code authentication templates allow you to send a one-time password or code along with a copy code button to your users.

Voice Call Templates

Voice Call Templates allow WhatsApp consumers to initiate a voice call directly by clicking the button in the template.

Call Permission Request Templates

Call Permission Request Templates allow you to create templates that enable WhatsApp users to grant consent for receiving calls.

Note: For all Marketing type templates, you can set a Time-to-Live (TTL) value to define the delivery window, only if you have onboarded the MM Lite API. TTL must be specified in seconds, with a minimum of 43,200 seconds (12 hours) and a maximum of 2,592,000 seconds (30 days). The recommended TTL value is 86,400 seconds (24 hours).

url: Refers to a previously created button that allows the customer to visit the URL generated by appending the text parameter to the predefined prefix URL in the template.
index	Number of button index starting from 0, only if quick reply or dynamic URL buttons have been registered, It can have up to three quick reply buttons or only one dynamic URL button.
example: 0
minimum: 0
maximum: 2

parameters => type	Parameter types are image, video, document,location, payload
image => link	URL of a image sent in the header, only if having media URL. It is expected to be a valid URL starting with https:// or http://. Supported image types are PNG, JPG, JPEG. Maximum image size is 5MB.
image => id	This is the id of a media, You can create id by uploading the image using the media APIs explained in this document. Use the media id fetched after uploading the media.
video => link	URL of a video sent in the header, only if having media URL. It is expected to be a valid URL starting with https:// or http://. Supported video types are MP4, 3GPP. Maximum video size is 16MB.
video => id	This is the id of a media, You can create id by uploading the video using the media APIs explained in this document. Use the media id fetched after uploading the media.
document => link	URL of a document sent in the header, only if having media URL. It is expected to be a valid URL starting with https:// or http://. Supported document type is PDF. Maximum document size is 100MB

document => filename	Add a file name to show in WhatsApp.
location => latitude	Lattitude of the location. For example: 23.0917101.
location => longitude	Longitude of the location. For example: 72.5949363.
location => name	Name of the place. For example: Fab India Store.
location => address	Detailed address of the location For example: Mumbai Metropolitan Region
payload	Payload of a quick reply button, only if quick reply buttons have been registered. It should be in text format. Add the keyword as payload.
source	Source can be a label or string that indicates the platform or third party name from where the message is sent.
metaData => custom_callback_data	custom_callback_data optional is field allows businesses to pass a custom tracking value which will be returned unprocessed in all subsequent webhook callbacks (sent, delivered, read, failed).

Validating Payloads
We sign all Event Notification payloads with a sha256 signature and include the signature in the request's x-hub-signature-256 header. You don't have to validate the payload, but you should.

To validate the payload:
Generate a sha256 signature using the payload and your partner token.
Compare your signature to the signature in the x-hub-signature-256 header.
If the signatures match, the payload is genuine.
